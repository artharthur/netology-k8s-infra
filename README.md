# Дипломный проект: облачная инфраструктура, Kubernetes, мониторинг и CI/CD

Проект по заданию должен был разворачиваться в Яндекс.Облаке, но из-за недоступности облака он целиком воспроизведён локально на Proxmox. Заменены только провайдер инфраструктуры и managed-сервисы; логика и состав работы соответствуют заданию. Всё лежит в одном репозитории на GitHub.

Ниже — по чек-листу сдачи: что сделано и где это находится.

---

## 1. Репозиторий с конфигурацией Terraform и создание ресурсов с нуля

Конфигурации Terraform разнесены на две папки, как требует задание:

- `terraform/bootstrap` — сервисный аккаунт (пользователь с урезанными правами, без прав суперпользователя) и S3-хранилище под стейт. Стейт бутстрапа локальный.
- `terraform/infra` — основная инфраструктура: ресурсный пул, сеть с тремя подсетями в разных зонах доступности, виртуальные машины кластера и вспомогательные узлы. Стейт хранится в S3-бакете (backend), созданном на этапе bootstrap.

`terraform apply` и `terraform destroy` выполняются без ручных действий. Создание всех ресурсов с нуля демонстрируется на защите.

Адаптация: роль S3 (Object Storage) выполняет Garage — S3-совместимое хранилище. VPC с подсетями в зонах реализована тремя изолированными сетевыми сегментами с NAT-шлюзом.

## 2. Terraform pipeline (CI/CD-terraform)

Вместо Atlantis / Terraform Cloud настроен автоматический запуск Terraform из CI: при коммите в папку `terraform/` запускается `terraform plan` и `terraform apply`.

- Конфигурация пайплайна: `.github/workflows/terraform.yml`
- Прогон пайплайна (plan/apply) — на скриншотах.

## 3. Репозиторий с конфигурацией Ansible

Kubernetes-кластер ставится через Ansible.

- Роли и плейбуки: `ansible/`
- Установка кластера: роль `ansible/roles/k3s` и плейбук `ansible/playbook-k3s.yml`
- Подготовка узлов, сеть, S3-хранилище, registry, CI-runner — отдельными ролями в `ansible/roles/`.

Кластер: три узла (1 control-plane + 2 worker), распределены по разным подсетям. После установки `kubectl get pods --all-namespaces` отрабатывает без ошибок, доступ к кластеру — через kubeconfig.

## 4. Репозиторий с Dockerfile тестового приложения и собранный image

- Приложение и Dockerfile: `app/` (nginx, отдающий статическую страницу).
- Образ собирается в CI и публикуется в registry.

Адаптация: используется собственный (локальный) Docker Registry, развёрнутый рядом с кластером, — внешний registry с моего канала недоступен по сети. Собранный образ в registry показан на скриншоте.

## 5. Репозиторий с конфигурацией Kubernetes-кластера

Манифесты для настройки кластера:

- `k8s/monitoring/` — Prometheus, Grafana, Alertmanager, node-exporter и дашборды кластера.
- `k8s/ingress-traefik.yaml`, `k8s/ingress-routes.yaml` — ingress-контроллер и маршруты на 80 порт.
- `k8s/app-deployment.yaml`, `k8s/app-service.yaml` — деплой и сервис тестового приложения.

## 6. Доступ к приложению и Grafana

Оба сервиса доступны по http на 80 порту через ingress.

- Тестовое приложение — отдаёт статическую страницу.
- Grafana — веб-интерфейс с дашбордами по состоянию кластера. Данные доступа: логин `admin`, пароль `admin`.

Доступ демонстрируется скриншотами (приложение и Grafana с дашбордами).

## 7. Один ресурс

Все конфигурации (Terraform, Ansible, Kubernetes, приложение, CI/CD) хранятся в одном репозитории на GitHub.

---

## CI/CD приложения

- Конфигурация: `.github/workflows/ci-cd.yml`
- При коммите в код приложения — сборка Docker-образа и публикация в registry.
- При создании тега (например, v1.0.0) — сборка образа с этой версией и деплой в кластер.
- Сборка и деплой выполняются на self-hosted runner (кластер во внутренней сети). Прогоны пайплайна — на скриншотах.

## Структура репозитория

```
terraform/
  bootstrap/   сервисный аккаунт + S3 (Garage) под стейт
  infra/       пул, сеть (зоны), узлы кластера, runner
ansible/       роли и плейбуки: сеть, S3, подготовка узлов, k3s, registry, runner
k8s/
  monitoring/  Prometheus, Grafana, Alertmanager, node-exporter, дашборды
  ingress-*    Traefik и маршруты на 80 порт
  app-*        деплой и сервис приложения
app/           nginx + Dockerfile + статика
.github/workflows/  CI/CD приложения и terraform pipeline
```

Секреты (токены, пароли, ключи доступа) в репозиторий не коммитятся — хранятся в GitHub Secrets для пайплайнов и в gitignore локально.
