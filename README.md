# Дипломный проект: облачная инфраструктура, Kubernetes, мониторинг и CI/CD

Проект по заданию разворачивался в Яндекс.Облаке, но из-за недоступности облака целиком воспроизведён локально на Proxmox. Заменены только провайдер инфраструктуры и managed-сервисы — логика и состав работы соответствуют заданию. Весь проект в одном репозитории на GitHub.

Ниже разобрано по пунктам сдачи: что сделано и где находится. Скриншоты-подтверждения — под соответствующими разделами.

---

## 1. Terraform и создание ресурсов с нуля

Конфигурации Terraform разнесены на две папки:

- `terraform/bootstrap` — сервисный аккаунт с урезанными правами (без прав суперпользователя) и S3-хранилище под стейт.
- `terraform/infra` — основная инфраструктура: ресурсный пул, сеть с тремя подсетями в разных зонах доступности, узлы кластера и вспомогательные машины. Стейт хранится в S3-бакете, созданном на этапе bootstrap.

`terraform apply` и `terraform destroy` проходят без ручных действий.

Роль S3 (Object Storage) выполняет Garage. VPC с подсетями в зонах реализована тремя изолированными сетевыми сегментами с NAT наружу.

*(скриншот: terraform apply + созданные ресурсы в Proxmox)*

## 2. Terraform pipeline

Автоматический запуск Terraform из CI: при коммите в `terraform/` выполняется `terraform plan` и `terraform apply`.

Конфигурация: `.github/workflows/terraform.yml`

*(скриншот: успешный прогон пайплайна, шаги plan/apply)*

## 3. Ansible

Kubernetes-кластер ставится через Ansible. Роли и плейбуки — в `ansible/`. Кластер из трёх узлов (1 control-plane + 2 worker), распределённых по разным подсетям. После установки `kubectl get pods --all-namespaces` отрабатывает без ошибок, доступ к кластеру — через kubeconfig.

*(скриншот: прогон плейбука k3s)*

## 4. Тестовое приложение и образ

Приложение и Dockerfile — в `app/` (nginx, отдаёт статическую страницу). Образ собирается в CI и публикуется в registry.

Registry — собственный, развёрнут рядом с кластером (внешний registry по сети недоступен).

*(скриншот: собранный образ в registry)*

## 5. Конфигурация Kubernetes-кластера

- `k8s/monitoring/` — Prometheus, Grafana, Alertmanager, node-exporter и дашборды кластера.
- `k8s/ingress-traefik.yaml`, `k8s/ingress-routes.yaml` — ingress и маршруты на 80 порт.
- `k8s/app-deployment.yaml`, `k8s/app-service.yaml` — деплой и сервис приложения.

## 6. Доступ к приложению и Grafana

Оба сервиса открываются по http на 80 порту через ingress: тестовое приложение со статической страницей и веб-интерфейс Grafana с дашбордами по состоянию кластера. Данные доступа к Grafana приложены отдельно.

*(скриншоты: страница приложения; Grafana с дашбордом кластера)*

## 7. Один ресурс

Terraform, Ansible, Kubernetes-манифесты, приложение и CI/CD — всё в одном репозитории на GitHub.

---

## CI/CD приложения

Конфигурация: `.github/workflows/ci-cd.yml`

При коммите в код приложения собирается Docker-образ и публикуется в registry. При создании тега (например v1.0.0) образ собирается с этой версией и деплоится в кластер. Сборка и деплой идут на self-hosted runner, так как кластер во внутренней сети.

*(скриншот: успешный прогон CI/CD по тегу)*

## Структура репозитория

```
terraform/
  bootstrap/   сервисный аккаунт + S3 (Garage) под стейт
  infra/       пул, сеть (зоны), узлы кластера, runner
ansible/       роли и плейбуки: сеть, S3, подготовка узлов, k3s, registry, runner
k8s/
  monitoring/  Prometheus, Grafana, Alertmanager, node-exporter, дашборды
  ingress-*    Traefik и маршруты на 80 порт
  app-*        деплой и сервис приложения
app/           nginx + Dockerfile + статика
.github/workflows/  CI/CD приложения и terraform pipeline
```
